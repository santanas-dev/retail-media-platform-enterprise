# ADR-019: Channel Orchestrator — Pragmatic Deferral

**Date:** 2026-07-20
**Status:** ACCEPTED
**Supersedes:** b4-channel-orchestrator-design-gate.md (now DEFERRED)

## Context

§24 ТЗ описывает мультиканальную архитектуру: channel-agnostic ядро + Channel
Orchestrator + адаптерный слой + mock-адаптеры. Design gate B.4 предложил строить
этот слой **до** реализации первого канала.

На 2026-07-18 выяснилось: EDGE-001/002/003/004 (onboarding, manifest+подпись, PoP,
heartbeat) уже реализованы и покрыты тестами как КСО-вертикаль. Рефакторинг 4
готовых компонент в «ядро + адаптер» без второго реального канала — это чистый
churn без проверяемой ценности.

## Decision

**Channel Orchestrator + адаптерный слой + mock-адаптеры откладываются до появления
второго реального канала.**

До этого момента:
- КСО-плеер (M2) строится напрямую на контрактах EDGE-001..004 с ТОНКИМ адаптерным
  швом (manifest.adapter_payload, нормализованная proof-модель — уже есть в EDGE-002).
- Универсальные контракты держим channel-agnostic **на уровне данных** (manifest
  schema, proof schema), но **не строим оркестрацию наперёд**.
- Любую абстракцию валидируем на КСО как единственном реальном канале.
- Никаких хуков под несуществующие каналы.

## Consequences

- **Позитивные:** нет рефакторинга проверенного кода; KSO-плеер быстрее доходит до
  пилота; архитектурный долг минимален (контракты уже channel-agnostic).
- **Риски:** при появлении 2-го канала потребуется выделить adapter interface из
  существующего manifest/proof-кода. Риск принят — стоимость рефакторинга одного
  адаптера значительно ниже стоимости предварительной оркестрации без второго канала.
- **Отклонённая альтернатива:** каркас каналов сначала (дизайн-гейт B.4, буквально
  по ТЗ §24/§25) — требовал рефакторинга EDGE-001..004 до появления второго канала.
