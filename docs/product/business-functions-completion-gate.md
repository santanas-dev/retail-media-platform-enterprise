## Executive Summary

**Вопрос:** «Почему мы подошли к написанию плеера, если много функций в статусе "🟡 готово для пилота" с ограничениями?»

**Краткий ответ:** Пользователь прав. Сверка с ТЗ v2.5 показывает, что часть ограничений в статусе «🟡 готово для пилота» — это не UX-polish, а реальные незакрытые требования ТЗ. Плеер нельзя начинать, пока бизнес-функции, от которых зависит жизнеспособность пилота, не доведены до уровня «✅ Готово» хотя бы в минимальном объёме первого ТЗ.

**Verdict:** ⛔ **NO-GO для player architecture.** Обнаружены BLOCKER BEFORE PLAYER ограничения:
- Нет прогноза доступности при создании кампании (ТЗ §6.3)
- Нет sold-out + альтернатив (ТЗ §6.3)
- Нет настройки правил инвентаря в админке (ТЗ §6.3)
- Нет симуляции перед публикацией (ТЗ §22.4)
- Нет дашборда кампании с план/факт и причинами отклонений (ТЗ §12)
- Emergency не применяется на устройствах (ТЗ §13)

---

## Почему «🟡 готово для пилота» ≠ «✅ Готово»

Статус «🟡 готово для пилота» означает: базовый функционал работает в тестовом окружении, но есть ограничения, которые не позволяют использовать функцию в production-пилоте. Для многих функций это были сознательные упрощения (deferred). Теперь, перед стартом плеера, нужно explicit decision: какие упрощения допустимы, а какие — нет.

Сверка с ТЗ v2.5 показывает, что часть ограничений — это требования ТЗ, которые были отложены, но не отменены.

## 3. Почему мы остановили player preparation

S-084 сказал «GO для архитектурной подготовки плеера» на основе технического анализа (minimum inventory contract, manifest, gateway). Но техническая готовность ≠ бизнес-готовность.

Сверка с ТЗ показала, что 6 бизнес-требований являются BLOCKER BEFORE PLAYER. Это не «UX polish» и не «можно отложить» — это требования ТЗ, без которых бизнес-процесс размещения рекламы неполон:

1. **Менеджер не видит прогноз доступности при создании кампании.** Он не знает, есть ли свободные слоты на выбранной поверхности. Кампания может быть создана, отправлена на согласование, и только потом выяснится что мест нет.

2. **Система не предлагает альтернативы при sold-out.** Менеджер должен вручную искать другие поверхности.

3. **Правила инвентаря нельзя настроить через UI.** Блэкауты, лимиты SOV, прайм-тайм — всё через БД. Оператор не может самостоятельно управлять инвентарём.

4. **Нет симуляции перед публикацией.** Кампания публикуется «вслепую» — без проверки конфликтов, охвата и влияния на другие кампании.

5. **Нет дашборда кампании с анализом отклонений.** Рекламодатель видит только голые цифры показов, без объяснения почему план не выполнен.

6. **Emergency не работает на устройствах.** Кнопка «стоп-реклама» есть в админке, но на КСО реклама продолжит показываться.

## 5. Recommendation

### ⛔ NO-GO для player architecture

**Причина:** 6 BLOCKER BEFORE PLAYER ограничений (A1–A6). Это не «можно отложить» — это требования ТЗ v2.5, без которых бизнес-процесс размещения рекламы неполон.

**Что нужно до player:** закрыть A1–A6 (S-086–S-091).

**Ожидаемый эффект:** после S-086–S-091:
- Менеджер видит доступность при создании кампании
- Система предлагает альтернативы при sold-out
- Оператор управляет правилами инвентаря через UI
- Перед публикацией видна симуляция
- Рекламодатель видит анализ отклонений
- Emergency реально останавливает рекламу

**После этого:** GO для player architecture.
