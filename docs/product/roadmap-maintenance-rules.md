# Правила ведения дорожной карты (Roadmap Maintenance Rules)

> Версия: 1.1
> Дата: 2026-07-16
> Автор: S-078b, S-079b

## Единый источник

**Один файл:** `docs/product/roadmap-s020-2026-07-10.xlsx`

Никаких копий, дубликатов в Google Sheets, выгрузок в Confluence или
альтернативных форматов. Вся дорожная карта живёт в одном xlsx-файле.

## Структура зафиксирована

- **Ровно 2 листа.** Третий лист не добавлять.
- **Названия листов не менять:**
  1. `Технический Roadmap`
  2. `Бизнес-функции Roadmap`
- **Технический Roadmap — 5 колонок:**
  1. Модуль / Подсистема
  2. Статус
  3. Коммиты / Тесты / ADR
  4. Следующий шаг
  5. Заметки
- **Бизнес-функции Roadmap — 8 колонок (S-079b):**
  1. Версия / этап
  2. Бизнес-функция
  3. Статус
  4. Для чего нужна
  5. Что уже можно использовать
  6. Что ещё нельзя / ограничения
  7. Почему сейчас / зависимость
  8. Следующий шаг
- Технический и бизнес-листы могут иметь разное число колонок — это нормально.
- **Не пересоздавать таблицу заново.** При обновлении статуса/описания —
  редактировать существующие ячейки, не перестраивать структуру.

## Язык

- **Бизнес-функции Roadmap:** русский язык. Описания должны быть понятны
  владельцу продукта без технического бэкграунда.
- **Технический Roadmap:** преимущественно русский. Технические названия
  (API endpoints, ADR, названия пакетов, имена технологий) допустимо
  оставлять на английском, если это имя собственное.
- **Статусы:** строго на русском по единому словарю (см. ниже).

## Единый словарь статусов

| Статус | Значение |
|--------|----------|
| ✅ Готово | Полностью реализовано, протестировано, в develop/main |
| 🟡 Готово для пилота | Работает для демо/пилота, но не для production |
| 🚧 В работе | Активно разрабатывается в текущем спринте |
| 🟠 Частично готово | Часть функциональности есть, часть — нет |
| ⚪ Не начато | Запланировано, но разработка не начиналась |
| ⏳ Запланировано | Есть в плане на конкретную версию |
| 🚫 Отложено | Сознательно отложено (не в приоритете) |
| 🔒 Заблокировано зависимостью | Не может начаться, пока не готова другая функция |
| 🧭 Требует решения | Нужно архитектурное или продуктовое решение |

**Правила применения:**
- Не повышать статус без фактического завершения работы.
- Не использовать английские эквиваленты (Done, Planned, Deferred, etc.).
- Не смешивать несколько статусов в одной ячейке (выбирать наиболее точный).

## Что обновлять

При каждом завершённом S-тикете или значимом изменении:

1. **Статус** соответствующей строки в Техническом Roadmap.
2. **Коммиты / Тесты / ADR** — актуальные цифры и ссылки.
3. **Следующий шаг** — что реально будет дальше.
4. **Заметки** — контекст, важный для понимания статуса.

Для бизнес-функций дополнительно:
5. **«Для чего нужна»** — бизнес-смысл функции, понятный владельцу продукта.
6. **«Что можно использовать сейчас»** — честно, без overclaim.
7. **«Что ещё нельзя / ограничения»** — с объяснением причины.
8. **«Почему сейчас / зависимость»** — почему эта функция делается именно на этом этапе, от чего зависит.
9. **«Следующий шаг»** — что реально будет дальше.
10. **Не заменять бизнес-смысл S-номерами.** S-номер можно указать в конце как ссылку, но не вместо описания.

## Теги и релизы

- Теги ставятся на **code baseline** (коммит с реальным кодом), не на
  docs-only коммиты.
- Релизная версия в теге должна соответствовать milestone в дорожной карте.
- После публикации тега — обновить milestone-строку в Техническом Roadmap.

## Синхронизация с PROJECT_STATE

- **Статус «✅ Готово» в дорожной карте требует того же behavioral proof,
  что и закрытие пункта в PROJECT_STATE.** Карта не может помечать пункт
  как done, если PROJECT_STATE фиксирует его как открытый Critical,
  not closed, merge pending или awaiting verification.
- При каждом обновлении статуса в дорожной карте проверять PROJECT_STATE.md:
  статусы в карте не должны быть выше, чем в PROJECT_STATE.
- Если карта противоречит PROJECT_STATE — править карту, не PROJECT_STATE.
  PROJECT_STATE — канонический источник истины о состоянии задач.
- После закрытия критической задачи (H0, C2, D1) проверять, что статус
  в дорожной карте обновлён и ссылается на SHA/CI run закрывающего коммита.

## Синхронизация с feature-registry и UI-smoke (UI-TRUTH-001B)

С **REGISTRY-EXPAND** (2026-07-18) введено правило консистентности:

**Бизнес-функцию в дорожной карте запрещено помечать «✅ Готово» или
«🟡 Готово для пилота», если соответствующие ей записи в
`docs/product/feature-registry.yaml` имеют статус `blocked`.**

Конкретно:
- Статус `✅ Готово` / `🟡 Готово для пилота` в колонке «Статус»
  Бизнес-функций Roadmap разрешён **только** если ВСЕ соответствующие
  функции feature-registry имеют `status: reachable` и зелёный UI-smoke
  (для UI-функций) или зелёный behavioral proof (для service-функций).
- Если хотя бы одна функция домена `blocked` — статус в дорожной карте
  должен быть не выше `🟠 Частично готово` с указанием причины в
  колонке «Что ещё нельзя».
- Service-функции (manifest, PoP, onboarding, heartbeat, observability)
  проверяются по behavioral CI, а не по UI-smoke.

**Автоматический сторож** (`scripts/roadmap-consistency-check.py`)
запускается в CI как non-blocking audit job. Он находит рассинхроны
между roadmap, registry и smoke-тестами. Режимы:

- `python3 scripts/roadmap-consistency-check.py` — audit mode (exit 0)
- `python3 scripts/roadmap-consistency-check.py --strict` — blocking mode
  (exit 1 при нарушениях, включается после reconcile)

**Порядок reconcile:**
1. Довести UI-smoke до зелёного для P0-журнеев (G1→G4).
2. Обновить feature-registry: `status: reachable` для закрытых журнеев.
3. Обновить roadmap: статусы на основе registry.
4. После полного reconcile — включить `--strict` в CI.
