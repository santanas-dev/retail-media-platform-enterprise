## 1.1. Назначение системы

Система «Центр управления рекламой» предназначена для централизованного управления рекламным и информационным контентом на цифровых носителях розничной сети: экранах касс самообслуживания (КСО), Android/Android TV устройствах, ТВ-приставках, Android price checker, электронных ценниках/ESL через шлюзы и vendor API, LED shelf banners и будущих типах retail media носителей. КСО является первым базовым каналом внедрения, но система должна проектироваться как самостоятельная мультиканальная платформа, изолированная от существующего Retail Media Dashboard и архитектурно готовая к интеграции с корпоративными системами отчётности, продаж, BI, кассовыми системами и системами управления ценниками.

## 1.2. Бизнес-цели

Единый контроль рекламного и информационного контента на КСО и других цифровых носителях по всей сети.

Планирование, продажа и контроль рекламного инвентаря: свободное и занятое эфирное время, прогноз показов и занятость слотов.

Централизованное управление рекламными кампаниями, креативами, размещениями и плейлистами.

Поддержка иерархии: Сеть → Филиал → Кластер → Магазин → Устройство.

Доказательная база показов для рекламодателей: Proof-of-Play, контроль версий, подписи, отчёты.

Автономная работа поддерживаемых устройств/плееров при потере связи с сервером с учётом возможностей конкретного типа носителя.

Безопасная эксплуатация внутри корпоративного контура без влияния на работу кассового ПО УКМ4, систем проверки цены, электронных ценников и иных критичных торговых процессов.

## 4.3. Размещение и сеть

Серверные компоненты размещаются в ЦОД компании внутри корпоративного контура.

Не допускается проектирование как единой плоской L2-сети для всех магазинов. Доступ устройств, плееров, шлюзов ESL/LED и Android-клиентов должен идти через маршрутизируемые сегменты с firewall-правилами.

Устройства и шлюзы должны инициировать исходящие подключения к Device Gateway по HTTPS/mTLS или через утверждённый vendor/API-адаптер. Входящие подключения из ЦОД к устройствам магазинов не являются обязательными и должны быть запрещены без отдельного обоснования.

Административный веб-интерфейс доступен только из корпоративной сети/VPN и через AD/SSO/MFA.

MinIO, PostgreSQL, ClickHouse и Redis не должны быть напрямую доступны магазинам и рабочим станциям пользователей.

Для emergency-команд должен быть отдельный высокоприоритетный канал доставки через Device Gateway.

## 6.1. Пользователи, роли и доступ

Поддержка AD/LDAP или корпоративного SSO. Локальная учётная запись допускается только как резервная административная запись.

RBAC: роли, права, группы, ограничения по филиалам/кластерам/магазинам/рекламодателям.

MFA обязательно для администраторов, согласующих и пользователей с правом публикации/emergency.

Запрет общих учётных записей. Все действия должны быть привязаны к конкретному пользователю.

RLS на уровне данных: рекламодатель видит только свои кампании, филиал — только свой контур, администратор — согласно назначенным правам.

## 6.2. Рекламодатели, заказы и кампании

Справочник рекламодателей и брендов: юридическое лицо, бренд, контакты, ответственные, договор/заказ.

Кампания должна быть бизнес-объектом: рекламодатель, период, цель, лимиты, бюджет/объём, статус, ответственные.

Размещение должно быть отдельным объектом: где, когда, какой креатив, какая частота, какой приоритет, какие ограничения.

Статусы кампании: draft → moderation → review → approved → scheduled → live → paused → completed → archived/cancelled.

Поддержка приостановки кампании, частичного снятия с магазинов, продления, изменения креатива через новую версию.

A/B-тестирование должно иметь правила: выбор контрольной/тестовой группы, период теста, метрика победителя, ручное подтверждение результата.

## 6.3. Рекламный инвентарь и прогноз

Система должна показывать доступное, занятое и зарезервированное эфирное время по сети, филиалу, кластеру, магазину и устройству.

При создании кампании менеджер должен видеть прогноз доступных показов по периоду, времени, географии и типу контента.

Конфликтом считается не только пересечение расписания, но и превышение допустимой доли рекламного времени, приоритетов или лимитов показов.

Должны поддерживаться статусы инвентаря: свободно, зарезервировано, продано, занято внутренней кампанией, занято emergency/fallback.

Система должна уметь показывать sold out и предлагать альтернативные магазины/периоды/время.

Необходима настройка правил инвентаря в админке: максимальная рекламная нагрузка, длительность слота, прайм-тайм, приоритеты, filler-контент.

| Любые исполняемые файлы | Запрещено | Не допускаются к загрузке и распространению на цифровые носители. |
| ESL-шаблоны | Только через утверждённую интеграцию | Статичные шаблоны/сегменты; цена и обязательные реквизиты берутся из master-системы цен, а не из рекламного портала. |
| LED shelf banner assets | Разрешено после теста устройства | Низкое разрешение, ограничения по цветам, яркости, FPS, длительности, размеру файла и протоколу контроллера. |
| Android interactive layouts | Ограниченно разрешено | Только для price checker/Android-устройств; реклама не должна блокировать основную функцию проверки цены/товара. |


## 7.1. Поддерживаемые форматы

При загрузке рассчитывается SHA-256, фиксируются размер, разрешение, длительность, MIME-type, автор и версия.

Должна выполняться антивирусная проверка и проверка структуры файла.

Для каждого креатива хранится история версий. Замена креатива в live-кампании создаёт новую версию и требует согласования.

Удаление опубликованного медиа не должно ломать исторические отчёты; физическое удаление допускается только после окончания срока хранения.

Превью должно показывать реальный вид на области 1440×1080 и поведение при скрытии рекламы из-за касания УКМ4.

## 8.2. Требования к manifest

КСО получает не произвольную HTML-страницу с токеном в URL, а подписанный manifest через Device Gateway.

Manifest должен содержать: manifest_id, device_id, store_id, playlist_version, valid_from, valid_to, media_files, sha256, duration, order/weight, priority, emergency_flag, fallback_rules, signature.

Плеер обязан проверить подпись manifest и SHA-256 каждого медиафайла до показа.

Manifest имеет версию. Любое изменение расписания, креатива, приоритета или emergency-состояния создаёт новую версию.

КСО отправляет подтверждение: manifest получен, проверен, применён, ошибка загрузки, недостаточно места, неверная подпись.

# 9. Плееры и runtime-агенты цифровых носителей

## 9.1. Техническое решение

Система должна поддерживать несколько типов runtime: КСО Linux/Chromium, Android/Android TV native player или WebView kiosk, Android price checker app module, а также адаптеры к ESL/LED шлюзам и контроллерам.

Для КСО Chromium запускается в kiosk-mode в области 1440×1080 в левой части экрана Full HD; для Android/TV и других носителей размеры, ориентация, зоны экрана и поддерживаемые форматы определяются capability profile устройства.

Для прямых устройств рекомендуется локальный wrapper/agent, который управляет регистрацией, токенами/сертификатом, загрузкой manifest, кэшем, логами, перезапуском плеера и watchdog; для ESL/LED используется адаптер к утверждённому шлюзу/контроллеру.

JWT/access-токен не должен передаваться в URL. Авторизация выполняется через mTLS и/или заголовок Authorization.

Для КСО при касании экрана или активности УКМ4 рекламная область скрывается на настраиваемый период, по умолчанию 30 секунд. Для Android price checker реклама показывается только в безопасных зонах или idle/screen saver режиме и не блокирует проверку цены.

Плеер/адаптер не должен мешать фискальным операциям, проверке цены, работе электронных ценников, кассовым процессам и штатной операционной деятельности магазина.

## 9.2. Автономность и fallback

Медиафайлы, шаблоны, параметры отображения и последний валидный manifest кэшируются локально, если это поддерживает конкретный тип устройства.

При потере связи устройство продолжает выполнять последний валидный manifest до истечения valid_to/offline TTL; для ESL/LED фиксируется последнее подтверждённое состояние шлюза/контроллера.

После истечения offline TTL плеер переходит на fallback/filler-контент, утверждённый компанией.

PoP, apply-ack, ошибки и heartbeat накапливаются локально или на шлюзе и отправляются пачками при восстановлении связи.

Локальный кэш должен иметь лимит размера и алгоритм очистки старых неиспользуемых файлов.

Должна быть защита от бесконечного показа устаревшей платной рекламы без фиксации риска в отчётах.

# 10. Управление устройствами

| Статус устройства | Значение |
| --- | --- |
| online | Устройство регулярно отправляет heartbeat и принимает manifest. |
| offline | Нет heartbeat дольше установленного порога. |

| error | Плеер не работает или не может применить manifest. |
| maintenance | Устройство временно исключено из расчёта SLA и показов. |
| revoked | Сертификат/токен отозван, устройство заблокировано. |


Регистрация устройства выполняется по одноразовому device_code, hardware_fingerprint и выдаче сертификата/ключа устройства.

Для каждого физического устройства, плеера, шлюза или логического носителя хранятся: магазин, зона/полка/место установки, тип устройства, capability profile, серийный номер/идентификатор, hardware_fingerprint при наличии, версия ОС/firmware, версия плеера/адаптера, IP/сегмент, статус, last_seen, текущий manifest, размер кэша, ошибки и связь с родительским шлюзом/контроллером.

Массовые операции должны иметь ограничение по ролям, подтверждение, аудит и staged rollout.

Любое изменение устройства должно фиксироваться в журнале аудита.

## 11.1. Требования к PoP

PoP фиксирует факт начала/окончания показа, device_id, store_id, campaign_id, placement_id, creative_id, media_id, manifest_id, timestamp, duration, sha256, результат показа и подпись устройства.

PoP отправляется не по одному событию, а пачками с идемпотентными ключами, чтобы исключить дубли при повторной доставке.

События подписываются ключом устройства или сопровождаются HMAC/подписью с проверкой на сервере.

Сервер проверяет подпись, формат, временные отклонения, соответствие manifest и валидность устройства.

При потере связи события сохраняются локально и отправляются в хронологическом порядке после восстановления связи.

Должны фиксироваться причины недопоказа: офлайн-устройство, отсутствует файл, неверный SHA-256, истёк manifest, emergency, ошибка плеера, касание/скрытие.

# 13. Emergency Management

Emergency-команда должна иметь приоритет выше обычных кампаний и плейлистов.

Доступ к emergency-функциям только у ограниченного круга пользователей с MFA и обязательным указанием причины.

Уровни применения: устройство, магазин, кластер, филиал, вся сеть.

Типы действий: стоп-реклама, замена на системное сообщение, запуск fallback, возврат к штатному manifest.

Каждое действие требует аудита: кто, когда, уровень, причина, список затронутых объектов, результат доставки.

Система должна показывать прогресс применения emergency-команды по онлайн-устройствам.

## 16.2. Интеграция с УКМ4 и чеками

Интеграция с данными о чеках нужна для оценки охвата: показ × количество чеков/транзакций в магазине/периоде.

Интеграция не должна влиять на работу кассы и фискальные операции.

На первом этапе допускается пакетная загрузка агрегированных чековых данных, далее — регулярная интеграция по утверждённому API/ETL.

В отчётах необходимо явно разделять фактические показы и оценочный охват по чекам.

| Риск | Почему важно | Что сделать до разработки |

| Влияние на УКМ4 | Касса является критичной системой магазина. | Провести тест на реальной КСО: касания, зависание Chromium, перезапуск, фискальные операции. |

# 21. Требования к реализации проекта AI-агентом Hermes

## 21.1. Статус Hermes в проекте

Hermes используется как инструмент разработки и AI-агент, который пишет код по настоящему ТЗ. Hermes, DeepSeek и любые LLM-модели не являются частью создаваемой платформы, не должны попадать в архитектуру продукта и не должны быть обязательной зависимостью для работы системы в production.

Разрабатываемая система должна быть самостоятельной: backend, frontend, плееры/runtime-клиенты, Device Gateway, Channel Adapter Layer, аналитика, отчёты, мониторинг, аудит и эксплуатационные механизмы должны работать без подключения Hermes или LLM.

## 21.2. Общий порядок работы Hermes

Каждый шаг разработки выполняется небольшими итерациями: один шаг — один проверяемый результат.

Перед изменением файлов Hermes обязан сначала проанализировать текущее состояние проекта и кратко описать, что будет сделано.

До внесения изменений Hermes должен перечислить файлы, которые планирует изменить, и цель изменения.

Крупные изменения, затрагивающие несколько модулей сразу, запрещены без отдельного согласования.

После каждого шага должны быть выполнены проверка, тесты/миграции/сборка и краткий отчёт о результате.

После успешной проверки изменения фиксируются в Git отдельным commit.

## 21.3. Запреты и ограничения для Hermes

Hermes запрещено без отдельного согласования:

менять утверждённый технологический стек;

изменять бизнес-логику, ролевую модель, модель данных, API-контракты и требования ИБ;

упрощать production-требования до демонстрационной MVP-версии;

отключать авторизацию, аудит, проверку прав, валидацию входных данных или проверки безопасности;

хранить пароли, токены, ключи, сертификаты и другие секреты в исходном коде или репозитории;

использовать внешние API, облачные сервисы, публичные CDN и сторонние SaaS без согласования;

изменять схему БД без миграции и без описания последствий;

удалять существующие файлы, модули, тесты и документацию без отдельного подтверждения;

писать временные заглушки вместо production-логики без явной пометки TODO и отдельного согласования;

оставлять неработающий код, падающую сборку или неприменяемые миграции.

## 21.4. Требования к структуре проекта

Структура проекта должна быть заранее зафиксирована и не должна хаотично меняться в ходе разработки. Рекомендуемая структура:

/backend — FastAPI, бизнес-логика, API, сервисы, миграции;

/frontend — React + TypeScript, административный интерфейс и личный кабинет рекламодателя;

/players — код плееров и runtime-клиентов по каналам: KSO player, Android player, price checker module, manifest-клиент, локальный кэш;

/infra — Docker Compose, nginx/reverse proxy, конфигурации окружений, скрипты развёртывания;

/docs — архитектура, API-контракты, эксплуатационные инструкции, решения и ограничения;

/tests — unit, integration, e2e, нагрузочные и приёмочные тесты;

/scripts — служебные скрипты разработки и эксплуатации.

## 21.5. API-first и контрактная разработка

Перед реализацией frontend для каждого модуля должны быть определены API-контракты: endpoint, метод, request/response, статусы ошибок, права доступа, пагинация, фильтрация и сортировка.

OpenAPI/Swagger должен актуализироваться вместе с backend-кодом.

Для критичных операций должны быть предусмотрены idempotency key, audit trail и понятные коды ошибок.

Device API, пользовательский API, аналитический API и emergency API должны быть логически разделены.

## 21.6. Обязательные тесты

Hermes обязан покрывать тестами не только технические endpoints, но и ключевую бизнес-логику. Обязательные зоны тестирования:

аутентификация, RBAC/RLS и ограничения доступа;

иерархия Сеть → Филиал → Кластер → Магазин → Устройство;

наследование и переопределение плейлистов;

проверка конфликтов расписаний и рекламного инвентаря;

расчёт свободного/занятого эфирного времени и sold out;

версионирование креативов, плейлистов и manifest;

валидация медиафайлов, SHA-256 и ограничения форматов;

регистрация устройств, выдача сертификатов/токенов и отзыв доступа;

проверка подписи manifest и контроль целостности медиа;

приём PoP пачками, идемпотентность и защита от дублей;

emergency-команды, аудит и staged rollout;

экспорт отчётов и разграничение данных рекламодателей.

## 21.7. Порядок реализации этапов Hermes

1. Создать каркас проекта, инфраструктуру локального запуска и базовую документацию.

6. Реализовать медиатеку: загрузка, проверка, MinIO, SHA-256, версии, модерация.

7. Реализовать кампании, рекламодателей, заказы, размещения и инвентарь.

какие файлы изменены;

как проверить результат;

какие команды запустить;

какие тесты пройдены;

## 21.10. Приоритет безопасности при генерации кода

Hermes должен рассматривать требования ИБ как обязательные, а не как дополнительные. Если между скоростью разработки и безопасностью возникает конфликт, приоритет отдаётся безопасности, аудиту, контролю доступа, предсказуемой архитектуре и возможности отката.

# 22. Дополнительные лучшие практики для production-портала

Данный раздел является обязательной частью требований. Его цель — превратить систему из технического инструмента показа рекламы в полноценную промышленную бизнес-платформу для продажи, контроля и доказательства оказания рекламной услуги на цифровых носителях розничной сети.

## 22.1. Бизнес-процесс размещения рекламы

В системе должен быть формализован полный жизненный цикл рекламного размещения: заявка → расчёт доступного инвентаря → коммерческое предложение → бронь → загрузка креатива → модерация → согласование → публикация → показ → отчёт → закрытие кампании.

Для каждого шага должны быть определены ответственные роли, допустимые действия, статусы, сроки выполнения и критерии перехода на следующий этап.

Система должна различать коммерческие кампании, внутренние кампании сети, компенсационные размещения, тестовые кампании и filler-контент.

Для кампании должны храниться бизнес-параметры: рекламодатель, бренд, заказ/договор, период, целевая география, плановый объём, приоритет, ответственный менеджер, статус оплаты/подтверждения при необходимости.

При изменении сроков, географии, креатива или объёма кампании должна создаваться новая версия размещения с аудитом и возможностью сравнения изменений.

Закрытие кампании допускается только после формирования итогового отчёта, фиксации план/факт и причин отклонений, если они есть.

## 22.2. SLA и качество оказания рекламной услуги

В системе должны быть зафиксированы показатели качества услуги, которые понятны бизнесу, ИТ и рекламодателю.

Доступность административного портала: целевое значение не ниже 99,5% в рабочее время, если иное не утверждено отдельно.

Доступность Device Gateway для цифровых устройств, плееров и шлюзов: целевое значение не ниже 99,9% внутри корпоративной сети.

Доставка нового manifest до онлайн-устройства: целевое значение до 5 минут для обычных изменений.

Доставка emergency-команды до онлайн-устройства: целевое значение до 1 минуты, с фиксацией устройств, которые команду не получили.

Потеря PoP-событий не должна превышать утверждённый порог. Целевое значение — не более 0,1% от ожидаемого объёма событий.

Доля активных устройств, логических носителей и поверхностей показа по каналам должна отображаться в отчёте рекламодателя и в отчёте качества размещения.

Максимальная задержка аналитического отчёта после поступления PoP-событий должна быть определена и контролироваться метриками.

## 22.3. Недопоказы, причины отклонений и компенсации

Система должна отдельно учитывать недопоказы и причины невыполнения плана кампании.

Недопоказом считается разница между плановым и фактическим объёмом показов, если фактический объём ниже допустимого порога выполнения.

Причины недопоказа должны классифицироваться: технические, операционные, бизнесовые, emergency, ошибка контента, ошибка планирования, недоступность устройства, недоступность магазина.

Типовые причины: устройство/канал офлайн, магазин закрыт, плеер или адаптер не запущен, медиа не скачалось, SHA-256 не совпал, manifest не применён, истёк valid_to, носитель вытеснен emergency-сообщением, кампания вытеснена более высоким приоритетом, ESL/LED-шлюз не подтвердил применение.

Система должна автоматически предлагать варианты докрутки: дополнительные дни, альтернативные магазины, другие временные слоты, компенсационный объём.

Компенсационные показы должны быть отдельным типом размещения и не должны смешиваться с коммерчески проданным инвентарём без явной маркировки.

В отчёте рекламодателя должен быть отдельный блок: план, факт, процент выполнения, недопоказ, причины, предложенная или выполненная компенсация.

## 22.4. Приоритеты рекламы, вытеснение и симуляция перед публикацией

Правила приоритетов должны быть формализованы и настраиваемы через административный интерфейс с версионированием.

Рекомендуемая базовая иерархия приоритетов: emergency-сообщения → обязательная реклама сети → федеральные платные кампании → филиальные кампании → локальные кампании магазина → filler-контент → тестовые кампании.

Система должна явно показывать, какая кампания вытесняет другую и почему.

Перед публикацией кампании должна выполняться симуляция: сколько магазинов будет задействовано, сколько показов ожидается, где не хватает инвентаря, какие конфликты есть, какие кампании будут затронуты.

Запрещается публиковать кампанию с критичными конфликтами без отдельного подтверждения уполномоченного пользователя.

Должна быть настройка overbooking, если бизнес решит разрешить продажу сверх доступного инвентаря. По умолчанию overbooking запрещён.

Все изменения правил приоритетов должны применяться с даты/времени изменения и не должны ретроспективно менять исторические отчёты.

## 22.5. Контроль качества креативов (Creative QA)

Перед публикацией креатив должен пройти автоматические и ручные проверки качества.

Автоматические проверки: формат, MIME-type, разрешение, вес файла, длительность видео, кодек, FPS, битрейт, отсутствие звука, соответствие зоне 1440×1080, контроль SHA-256.

Визуальные проверки: читаемость текста, безопасные отступы, отсутствие мелкого шрифта, корректный вид на 75% экрана, отсутствие перекрытия рабочей области УКМ4.

Безопасность: антивирусная проверка, запрет исполняемых файлов, проверка архивов, запрет внешних сетевых обращений из HTML5-контента, CSP/sandbox.

HTML5/JS-контент на первом релизе должен быть либо запрещён, либо разрешён только в ограниченной песочнице после отдельного решения ИБ.

Результат проверки креатива должен храниться в системе: кто проверил, когда, какие проверки пройдены, какие замечания, какая версия креатива допущена к публикации.

## 22.6. Операционный центр здоровья цифровых носителей

В системе должен быть отдельный интерфейс для эксплуатации и ИТ-поддержки, не смешанный с коммерческими отчётами рекламного менеджера.

Операционный дашборд должен показывать: всего устройств/логических носителей/поверхностей по каналам, онлайн, офлайн, ошибка плеера/адаптера, ошибка контента, ошибка PoP/apply-ack, старая версия runtime/adapter, не применён manifest, недостаточно места на диске, критичные магазины и каналы.

Должна быть детализация до филиала, кластера, магазина, канала, устройства, логического носителя и поверхности показа.

Для каждого устройства должны отображаться: последний heartbeat, текущий manifest, версия плеера, версия Chromium, свободное место, размер кэша, последние ошибки, последний успешный показ, последние PoP-события.

Система должна уметь формировать список магазинов, полностью выпавших из рекламной сети, и список устройств, влияющих на выполнение активных кампаний.

Операционные проблемы должны влиять на прогноз выполнения кампании и на отчёт о недопоказах.

## 22.7. Staged rollout и rollback

Все изменения, влияющие на работу устройств и показ рекламы, должны выкатываться поэтапно.

Рекомендуемая схема rollout: лаборатория → 5 магазинов → 50 магазинов → 300 магазинов → 10% сети → 50% сети → вся сеть.

Rollout применяется к новым версиям плеера, форматам manifest, новым правилам расписания, новым типам креативов, массовым настройкам и потенциально рискованным изменениям.

На каждом этапе должны контролироваться метрики ошибок, PoP, heartbeat, применения manifest, загрузки медиа, аварий плеера.

Если порог ошибок превышен, rollout автоматически останавливается и требует решения ответственного пользователя.

Для каждой версии плеера, manifest-формата и критичной настройки должен быть предусмотрен rollback на предыдущую стабильную версию.

## 22.8. Версионирование объектов и неизменяемость истории

Любое изменение, которое влияет на показ рекламы или отчётность, должно иметь версию и быть воспроизводимым.

Версионируются: кампания, размещение, креатив, плейлист, manifest, правила приоритета, настройки инвентаря, настройки плеера, API-контракты, схема БД, отчёт рекламодателя.

Исторические отчёты должны строиться по тем версиям правил, manifest и креативов, которые действовали в момент показа.

Изменение креатива в активной кампании не перезаписывает старый креатив, а создаёт новую версию.

Удаление объектов, участвовавших в показах, должно быть логическим. Физическое удаление допускается только после истечения срока хранения и при отсутствии юридических/договорных ограничений.

Должна быть возможность посмотреть diff: что изменилось в кампании, размещении, плейлисте, правилах приоритета или manifest.

## 22.9. Feature flags

Для безопасного развития системы должны использоваться feature flags.

Feature flags позволяют включать новые функции на пилотные магазины, филиалы, роли или ограниченный процент устройств.

Примеры флагов: новый алгоритм расписания, новый формат PoP, HTML5-контент, новый плеер, новый отчёт, новый способ расчёта инвентаря, A/B-тесты.

Изменение feature flag должно фиксироваться в аудите: кто изменил, что изменил, на какой контур, когда, причина.

Для критичных feature flags должен быть быстрый rollback через административный интерфейс.

## 22.10. Наблюдаемость: логи, метрики, трассировка и алерты

Система должна быть наблюдаемой для ИТ, ИБ и бизнеса.

Backend, Device Gateway, PoP ingestion, Analytics, MinIO, PostgreSQL, ClickHouse, Redis и плеер должны отдавать структурированные логи и метрики.

Для каждой операции должен использоваться correlation_id/trace_id, позволяющий связать действие пользователя, генерацию manifest, доставку на устройство/адаптер/поверхность, proof-событие и отчёт.

Ключевые метрики: время генерации manifest, задержка heartbeat, скорость приёма PoP, ошибки PoP, ошибки загрузки медиа, ошибки применения manifest, риск недопоказа, время доставки emergency-команды.

Должны быть алерты по критичным событиям: массовый уход устройств/каналов в офлайн, рост ошибок PoP/apply-ack, недоступность Device Gateway, сбои адаптеров, переполнение дисков, деградация ClickHouse, сбои MinIO, нарушение SLA кампании.

Должны быть отдельные дашборды: технический, бизнесовый, ИБ, дашборд качества рекламной услуги.

## 22.11. Data Governance и сроки хранения данных

Система должна иметь утверждённые правила владения, доступа, хранения, архивации и удаления данных.

PoP-логи в горячем аналитическом хранилище: рекомендуемый срок 12–18 месяцев, точное значение утверждается бизнесом и юристами.

Архив PoP и итоговые отчёты: рекомендуемый срок 3–5 лет при наличии договорных споров или требований рекламодателей.

Аудит действий пользователей и системных изменений: рекомендуемый срок не менее 3 лет.

Системные технические логи: рекомендуемый срок 90–180 дней, если нет требований ИБ или расследования инцидента.

Креативы завершённых кампаний: рекомендуемый срок 1–3 года или согласно договорным требованиям.

Выгрузка данных должна быть ограничена правами, журналироваться и иметь ограничения по объёму и периоду.

Данные рекламодателей должны быть изолированы на уровне прав доступа и отчётности.

## 22.12. Финансовые сущности, тарифы и коммерческий учёт

Если система используется для продажи рекламного инвентаря, в ней должны быть предусмотрены базовые коммерческие сущности, даже если биллинг и ЭДО находятся в других системах.

Необходимые сущности: рекламодатель, бренд, договор, заказ, тариф, прайс-лист, скидка, пакет размещения, бонусные показы, компенсационные показы, статус оплаты/подтверждения при необходимости.

Кампания должна быть связана с заказом или основанием размещения: коммерческим, внутренним, компенсационным, тестовым.

Система должна показывать выручку/плановую стоимость по кампании, филиалу, магазину, периоду и типу инвентаря, если эти данные разрешено хранить в портале.

Должна быть возможность отделить платный инвентарь от внутренней рекламы сети и filler-контента.

История изменения тарифов должна версионироваться и не должна менять расчёты завершённых кампаний задним числом.

## 22.13. UX для разных ролей

Интерфейс должен быть спроектирован под задачи конкретных ролей, а не как единая универсальная админка.

Рекламный менеджер: создать кампанию, увидеть свободный инвентарь, проверить конфликты, загрузить креатив, отправить на согласование.

Согласующий: увидеть, что запускается, где запускается, какие риски, что изменилось, какие кампании будут затронуты, принять решение approve/reject.

Аналитик: план/факт, недопоказы, причины отклонений, выгрузки, сравнение кампаний, эффективность по магазинам и периодам.

ИТ/эксплуатация: здоровье устройств, ошибки, версии, rollout, rollback, диагностика, применённые manifest, технические инциденты.

ИБ: аудит действий, изменения прав, операции с устройствами, загрузка подозрительных файлов, emergency-действия, события безопасности.

Рекламодатель: только свои кампании, понятные отчёты, план/факт, карта/география, исключения, выгрузки PDF/XLSX/CSV без технического шума.

## 22.14. Тестовые данные и демонстрационный стенд

Для проверки качества разработки должен быть создан демонстрационный стенд с реалистичными тестовыми данными.

Минимальный набор seed-данных: 10 филиалов, 50 кластеров, 500 магазинов, 2 000 КСО, 500 Android/TV устройств, 300 Android price checker, 50 ESL-шлюзов с тестовыми логическими ценниками, 100 LED shelf banners, 20 рекламодателей, 100 кампаний, 500 креативов, не менее 1 000 000 тестовых PoP/apply-ack событий.

Данные должны покрывать разные сценарии: активные кампании, завершённые кампании, недопоказы, офлайн-устройства, emergency, конфликты расписания, компенсационные показы, sold out.

Демо-стенд должен позволять быстро проверять производительность фильтров, отчётов, прав доступа, UI и корректность бизнес-логики.

Тестовые данные не должны содержать реальные персональные данные, реальные токены, реальные договоры и секреты.

## 22.15. Нагрузочные профили и критерии производительности

Нагрузочное тестирование должно выполняться по заранее утверждённым профилям, близким к реальной эксплуатации.

Базовый профиль: 40 000 КСО, heartbeat каждые 30 секунд с jitter, запрос manifest каждые 30 секунд с HTTP 304/ETag, отправка PoP пачками каждые 60 секунд; дополнительный профиль должен моделировать Android/TV, price checker, ESL/LED шлюзы и логические поверхности с их собственными интервалами синхронизации.

Профиль массовой публикации: изменение кампании или emergency-команды на всю сеть с контролем времени генерации и доставки manifest.

Профиль аналитики: отчёты по крупной кампании за период 1 день, 7 дней, 30 дней, 12 месяцев.

Профиль личного кабинета рекламодателя: одновременная работа нескольких сотен внешних пользователей с read-only отчётами.

Профиль админки: до 100 одновременных внутренних пользователей с фильтрами, созданием кампаний, модерацией и просмотром устройств.

Для каждого профиля должны быть определены допустимые времена ответа API, время построения отчёта, допустимый процент ошибок, использование CPU/RAM/диска и поведение при деградации компонентов.

## 22.16. Mini-design перед каждым шагом разработки Hermes

Перед каждым изменением кода Hermes обязан сформировать mini-design и только после подтверждения приступать к изменению файлов, если это предусмотрено процессом работы.

Mini-design должен содержать: что реализуется, какие файлы изучены, какие файлы будут изменены, какие таблицы/эндпоинты затрагиваются, какие риски есть, как будет выполнена проверка, какой Git commit будет сделан.

Hermes не должен объединять в один шаг изменения разных слоёв без необходимости. Предпочтительный размер шага: один небольшой логический результат.

Если в процессе шага Hermes обнаруживает, что требуется изменить архитектуру, модель данных, права, безопасность, API-контракт или бизнес-логику, он обязан остановиться и запросить отдельное согласование.

В отчёте после шага Hermes должен отдельно указывать: что сделано, что не сделано, какие файлы изменены, как проверить, какие тесты запущены, результат тестов, команда Git commit.

## 22.17. Даты, время, часовые пояса и календарь показов

Так как сеть распределена по России, система должна корректно работать с часовыми поясами и календарями.

Все события в БД и логах должны храниться в UTC с фиксацией локального часового пояса магазина/устройства для отображения в UI.

Расписание кампаний должно задаваться в локальном времени магазина, если не указано иное бизнес-правило.

Отчёты должны явно показывать, в каком часовом поясе построены данные.

Переходы дат, ночные интервалы, праздничные дни, закрытие магазина и исключения расписания должны быть поддержаны как отдельные правила.

## 22.18. API versioning и совместимость устройств

Device API и внутренние API должны поддерживать управляемое развитие без массовой поломки устройств.

Device API должен иметь версию. Старые версии API должны поддерживаться в течение утверждённого срока, пока на сети остаются старые версии плеера.

Изменение формата manifest должно быть обратно совместимым либо выкатываться через staged rollout.

Плеер при heartbeat должен передавать поддерживаемые версии API/manifest, чтобы сервер мог отдать совместимый формат.

Ломающие изменения API допускаются только через отдельный план миграции, тестовый стенд и rollback-сценарий.

## 22.19. Доступность интерфейса и качество пользовательского опыта

Портал должен быть удобен для регулярной операционной работы, а не только для демонстрации.

Ключевые таблицы должны иметь фильтры, сохранённые представления, поиск, сортировки, экспорт, массовые действия и быстрый переход к детализации.

Для долгих операций должен отображаться прогресс: загрузка медиа, генерация отчёта, массовая публикация, rollout, экспорт.

Ошибки должны быть понятны пользователю: что произошло, почему, что сделать дальше, к кому обратиться.

В интерфейсе согласования обязательно показывать последствия действия: какие магазины/кампании/плейлисты будут затронуты.

Для рекламодателя интерфейс должен быть read-only, простым и без технических терминов, не нужных для проверки оказания услуги.

## 23.1. Общий принцип

Платформа должна проектироваться как channel-agnostic retail media система. КСО является первым каналом внедрения, но бизнес-логика кампаний, инвентаря, модерации, согласования, отчётности и PoP не должна быть жёстко привязана к КСО.

Для каждого типа носителя задаётся capability profile: разрешение, ориентация, зоны экрана, поддерживаемые форматы, максимальный вес и длительность контента, наличие видео/анимации/интерактивности, способ связи, режим кэша, тип PoP и протокол управления.

Ядро системы работает с универсальными сущностями: кампания, размещение, креатив, рендер креатива, канал, поверхность показа, manifest, событие показа/применения. Конкретная доставка реализуется через адаптер канала.

Новые типы носителей должны подключаться через адаптерный слой без переписывания ядра кампаний, инвентаря, отчётности и RBAC.

## 23.2. Типы носителей и особенности управления

## 23.3. Физическое устройство, логический носитель и поверхность показа

Система должна различать физическое устройство, логический носитель и поверхность показа. Один ESL-шлюз или LED-контроллер может управлять множеством логических ценников/баннеров в магазине.

Для логической поверхности должны храниться: магазин, торговая зона, полка/категория/SKU при необходимости, тип поверхности, размер/разрешение, ориентация, активность, связанный контроллер/шлюз, текущий manifest и статус последнего применения.

Для расчёта инвентаря и отчётности кампания должна уметь таргетироваться не только на магазин/кластер/филиал, но и на тип носителя, торговую зону, категорию, группу SKU, полку или набор логических поверхностей.

## 23.4. Рендеры креативов под разные каналы

Один креатив должен иметь несколько renditions: например КСО 1440×1080, TV full HD, вертикальный Android-экран, карточка price checker, ESL-шаблон, LED-низкоразрешённый ролик.

Кампания не может быть опубликована на канал, если для него отсутствует утверждённый rendition или явно не задан fallback.

Creative QA выполняется отдельно по каждому rendition: формат, разрешение, длительность, вес, читаемость, цветность, безопасность, соответствие ограничениям канала.

В предпросмотре кампании пользователь должен видеть, как креатив выглядит на каждом выбранном типе носителя.

## 23.5. Manifest и адаптеры каналов

Manifest должен быть универсальным, но расширяемым: общие поля едины для всех устройств, а channel-specific параметры передаются в отдельном adapter_payload.

Адаптер канала отвечает за преобразование manifest в команды конкретного плеера, Android-приложения, ESL-шлюза или LED-контроллера.

Адаптер не должен обращаться напрямую к PostgreSQL, ClickHouse, MinIO или внутренним таблицам. Все операции выполняются через утверждённые внутренние API и Device/Channel Gateway.

Для каждого адаптера должны быть реализованы health-check, логирование, retry, idempotency, circuit breaker, аудит и тестовый mock-режим.

## 23.6. Инвентарь и SLA по каналам

Инвентарь должен рассчитываться отдельно по каждому типу носителя. Эфирное время на КСО/TV, idle-показы на price checker, статичное размещение на ESL и цикл LED-баннера имеют разные единицы измерения и не должны смешиваться без нормализации.

Для каждого канала задаются собственные SLA: доступность устройств, скорость применения manifest, допустимая задержка PoP/apply-ack, срок обновления контента, правила недопоказа и компенсаций.

Отчёт рекламодателя должен показывать план/факт не только по кампании, но и по каналам: КСО, Android/TV, price checker, ESL, LED и другие.

## 23.7. Особые требования к электронным ценникам и price checker

Портал не должен становиться master-системой цены. Все price-related данные для ESL и price checker должны приходить из утверждённой системы ценообразования/товарного учёта или проходить валидацию с ней.

Нельзя допускать рассинхрон между ценником, price checker и кассовой ценой. Любые рекламные бейджи, QR, промо-сообщения и шаблоны должны быть отделены от обязательных price fields либо валидироваться до публикации.

Для price checker реклама допустима только в режимах, не мешающих основной функции: idle screen, заставка, боковая/нижняя зона, завершение сценария проверки цены. Основной сценарий покупателя всегда имеет приоритет.

Для ESL запуск канала в production допускается только после отдельного пилота с владельцем процесса ценников, ИБ, операционным блоком и юридической оценкой рисков.

## 23.8. Интерфейс портала для мультиканального управления

При создании кампании пользователь выбирает каналы размещения, типы устройств, географию, торговые зоны/категории/SKU при необходимости и видит ограничения каждого канала.

Система должна показывать матрицу готовности кампании: выбран канал, есть утверждённый rendition, хватает инвентаря, нет конфликтов, есть прогноз охвата, есть PoP-режим, SLA достижим.

В устройственном разделе должны быть фильтры по типу носителя, версии плеера/адаптера, магазину, зоне, статусу, текущему manifest и ошибкам применения.

В отчётности должны быть отдельные разрезы по каналам, устройствам, логическим поверхностям и причинам недопоказа/неприменения.

## 24.1. Главное архитектурное решение

Архитектурный принцип v2.5: channel-agnostic core + channel-specific adapters. Ядро системы не содержит логики, завязанной на КСО, Android, ESL, LED или конкретного производителя. Все различия каналов выносятся в профили возможностей, adapter layer, runtime/player и channel-specific payload.

## 24.2. Целевая логическая схема

Целевая архитектура должна состоять из следующих слоёв:

Концептуальный поток работы: пользователь создаёт кампанию → ядро проверяет права, креативы, инвентарь и согласования → Channel Orchestrator рассчитывает поверхности и версии заданий → адаптеры каналов доставляют manifest/команды в нужном формате → устройства или шлюзы возвращают proof/ack/status → аналитика строит план/факт, SLA и отчёты рекламодателя.

## 24.5. Channel Orchestrator

Channel Orchestrator — обязательный компонент между бизнес-ядром и адаптерами каналов. Он отвечает за перевод утверждённого размещения в конкретные задания для устройств и поверхностей.

рассчитывает целевые поверхности по правилам: география, канал, зона, категория, SKU, capability profile;

выполняет симуляцию перед публикацией: инвентарь, конфликты, вытеснения, риски недопоказа;

собирает channel-neutral manifest и channel-specific adapter_payload;

создаёт версии manifest и не перезаписывает историю;

управляет staged rollout и rollback по каналам, магазинам и группам устройств;

публикует задания в очередь/шину событий для адаптеров;

собирает статусы применения, ошибки, proof-события и передаёт их в аналитику.

## 24.6. Channel Adapter Layer

Адаптер канала — изолированный компонент, который знает особенности конкретного типа устройства, плеера, шлюза или vendor API. Адаптер не должен обращаться напрямую к таблицам PostgreSQL, ClickHouse, MinIO или секретам системы, минуя утверждённые API и очереди.

Для каждого адаптера должны быть описаны контракт, входной payload, выходные события, коды ошибок, SLA, retry-policy, circuit breaker, mock-режим и критерии приёмки.

## 24.7. Универсальный manifest и channel-specific payload

Manifest становится универсальным контрактом между платформой и каналами. Он содержит общую часть, одинаковую для всех устройств, и расширяемую часть adapter_payload для конкретного канала.

Запрещается встраивать access token/JWT в URL manifest или медиа. Доступ к медиа и заданиям должен контролироваться через Device/Channel Gateway, подписанные URL с коротким сроком жизни или mTLS в зависимости от утверждённой схемы.

## 24.8. Нормализованная Proof-модель

В отчётности нельзя смешивать разные типы доказательств как одинаковые «показы». Система должна хранить proof_type и явно объяснять рекламодателю, что именно подтверждено.

Для коммерческих отчётов необходимо разделять: плановые показы, фактические playback-события, подтверждения применения, доставленные команды, недопоказы и события с ошибкой. Итоговый отчёт должен показывать метрику по каждому каналу отдельно.

## 24.9. Event-driven архитектура доставки и событий

Для мультиканальности прямые синхронные вызовы «портал → устройство/вендор» не должны быть основным механизмом массовой публикации. Требуется событийная модель с очередями, повторными попытками и идемпотентностью.

Выбор конкретной технологии очереди утверждается отдельно. Redis допустим для простых задач и кэша, но архитектура должна позволять перейти на RabbitMQ, NATS, Kafka/Redpanda или другой брокер без переписывания бизнес-ядра.

## 24.10. Изменения в модели данных

Модель данных должна быть расширена так, чтобы новые каналы подключались добавлением справочников, профилей и адаптеров, а не изменением бизнес-таблиц кампаний.

## 24.11. API и совместимость

User API, Device API, Channel Adapter API и Internal Worker API должны быть логически разделены и версионированы.

Device/Channel API должен поддерживать negotiation: device/runtime сообщает поддерживаемые версии API, manifest schema и capability profile.

Ломающие изменения manifest или Device API допускаются только через staged rollout, совместимую версию и rollback.

Адаптеры должны работать через контракт: получить задание, подтвердить доставку, вернуть proof/ack/error, отдать health/status.

Для сторонних vendor API должны быть отдельные connector credentials, rate limit, журналирование, retry и circuit breaker.

## 24.12. Как это влияет на реализацию Hermes

Hermes должен начинать разработку не с плеера КСО, а с архитектурного каркаса мультиканальной платформы: справочники каналов, capability profiles, универсальная модель устройств/поверхностей, контракты manifest/proof и только затем первый KSO Adapter.

Запрещается реализовывать КСО как отдельную изолированную вертикаль, которую потом придётся переписывать под Android/ESL/LED. Любые таблицы, API и сервисы, названные только kso_* и не имеющие обобщённой модели канала/поверхности, должны быть отдельно согласованы.

## 24.13. Критерии приёмки архитектуры v2.5

В БД есть универсальные сущности каналов, устройств, логических носителей, поверхностей и capability profiles.

Кампания может таргетироваться не только на магазин, но и на канал, тип носителя, торговую зону, категорию/SKU и набор поверхностей.

Публикация кампании проходит через Channel Orchestrator и Adapter Layer, а не напрямую из campaign service в плеер/устройство.

Manifest имеет общую часть и adapter_payload, подписывается, версионируется и проверяется на совместимость.

Proof-события нормализованы и различают real playback, delivery ack, apply ack, label ack, controller ack и error/not_applied.

Для ESL и price checker портал не является источником цены и не может публиковать price-related данные без сверки с master-системой.

Добавление нового канала требует нового adapter/capability profile/renditions, но не требует переписывания кампаний, инвентаря, RBAC и отчётности.
